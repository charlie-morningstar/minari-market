## 그림 1. 시스템 구조도

"요청이 들어와서 어디를 거쳐 데이터까지 가는지"를 위에서 아래로 그리는 그림입니다.

```
   ┌──────────────────────────┐
   │   사용자 브라우저 / 휴대폰   │
   └──────────────────────────┘
              │  (HTTP 요청 / 채팅은 WebSocket)
              ▼
   ┌──────────────────────────────────────────────┐
   │                Flask 서버                       │
   │  ┌──────┬──────┬──────────┬──────┬────────┐   │
   │  │ 회원  │ 상품  │커뮤니티   │ 지갑  │ 관리자  │   │  ← 기능별 모듈
   │  │(auth)│(prod)│(채팅/신고)│(송금) │(admin) │   │
   │  └──────┴──────┴──────────┴──────┴────────┘   │
   │  ┌────────────────────────────────────────┐   │
   │  │ security.py : 로그인·권한확인·입력검증·해시 │   │  ← 보안 공통
   │  └────────────────────────────────────────┘   │
   │  ┌────────────────────────────────────────┐   │
   │  │ db.py : 파라미터 바인딩 쿼리               │   │
   │  └────────────────────────────────────────┘   │
   └──────────────────────────────────────────────┘
              │
              ▼
        ┌───────────────┐
        │ SQLite (DB파일) │
        └───────────────┘
```

- 맨 위 "사용자", 가운데 큰 상자 "Flask 서버", 맨 아래 "SQLite" 3층으로 그립니다.
- 큰 상자 안에 기능 모듈 5개(회원·상품·커뮤니티·지갑·관리자)를 나란히,
  그 아래에 `security.py`, `db.py` 두 줄을 넣어 주세요. "모든 기능이 보안 공통을
  거친다"는 느낌이 나면 됩니다.
- 오른쪽 옆에 작게 "채팅은 events.py(실시간 소켓)로 처리"라고 화살표로 덧붙이면 좋아요.

## 그림 2. 데이터베이스 ERD (표들의 관계)

테이블 5개를 상자로 그리고, 누가 누구와 연결되는지 선으로 잇는 그림입니다.
각 상자 맨 위에 표 이름, 그 아래에 주요 칸(컬럼)을 적습니다. `PK`는 대표키,
`FK`는 다른 표를 가리키는 칸이라는 표시예요.

```
        ┌────────── users ──────────┐
        │ id (PK)                    │
        │ username / password_hash   │
        │ balance / is_admin         │
        │ is_dormant / report_count  │
        └───────────────────────────┘
          │1        │1        │1     │1
          │N        │N        │N     │N(보내는/받는 2개 선)
          ▼         ▼         ▼      ▼
 ┌── products ──┐ ┌ reports ┐ ┌ messages ┐ ┌ transfers ┐
 │ id (PK)      │ │ id (PK) │ │ id (PK)   │ │ id (PK)    │
 │ title/price  │ │reporter │ │ room      │ │ sender_id  │
 │ seller_id(FK)│ │ _id(FK) │ │sender_id  │ │(FK)        │
 │ is_blocked   │ │target_ty│ │ (FK)      │ │receiver_id │
 │ report_count │ │pe/target│ │ content   │ │(FK)/amount │
 └──────────────┘ └─────────┘ └───────────┘ └────────────┘
```

- **users를 가운데(또는 위)에 크게** 두고, 나머지 4개를 아래에 배치하세요.
- 연결선에 "1 —— N"이라고 적습니다. 뜻은 "사용자 1명이 상품·신고·메시지를
  여러 개 가진다"입니다.
- transfers는 users와 **선을 두 개**(보내는 사람 / 받는 사람) 그으면 정확합니다.

## 그림 3. 화면 이동 흐름도

사용자가 화면을 어떻게 오가는지, 그리고 "로그인해야만 되는 것"이 무엇인지
보여주는 그림입니다.

```
        ┌────────────────────────┐
        │ 메인 = 상품 목록 / 검색   │  (로그인 안 해도 볼 수 있음)
        └────────────────────────┘
           │ 상품 클릭        │ 회원가입 / 로그인
           ▼                 ▼
     ┌──────────┐      ┌──────────┐
     │ 상품 상세  │      │  로그인   │
     └──────────┘      └────┬─────┘
                            │ (로그인 성공하면 아래 기능이 열림)
      ┌───────────┬─────────┼──────────┬─────────┐
      ▼           ▼         ▼          ▼         ▼
 ┌────────┐ ┌────────┐ ┌───────┐ ┌────────┐ ┌────────┐
 │ 상품등록 │ │전체채팅 │ │1:1쪽지 │ │지갑/송금│ │마이페이지│
 └────────┘ └────────┘ └───────┘ └────────┘ └────────┘
      (상품/프로필에서 "신고"도 로그인 후 가능)

 ┌──────────┐
 │ 관리자화면 │  ← 관리자 계정만 (점선으로 따로 표시)
 └──────────┘
```

- 위쪽 "메인(상품목록)"에서 시작해서, 로그인을 기점으로 기능들이 열리는 구조로
  그리면 됩니다.
- 로그인 뒤에만 되는 화면들(등록·채팅·쪽지·송금·마이페이지·신고)을 **점선 상자
  하나로 크게 묶고 "로그인 필요"**라고 적어두면 접근 권한이 한눈에 보입니다.
- 관리자 화면은 따로 떼서 "관리자만"이라고 표시하세요.
